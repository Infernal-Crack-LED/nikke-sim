# Archetype tags — NIKKE kit-role tagging for Team Builder filtering

> SSOT for the archetype tag system. The data lives in `data/archetype-tags.json`
> and is generated by `scripts/build-archetype-tags.ts`. This doc defines the
> vocabulary, how tags are derived, and how to re-run the sweep or tag a single
> new unit.

## Purpose

The Team Builder already filters by **stat axes** — weapon, burst, class,
element, manufacturer. Archetype tags add a **kit-role axis**: what a unit's kit
_does_ for the team. "Show me every reload buffer" or "every unit that deals
distributed damage" is a team-building question the stat axes can't answer.

Tags are **facets, not a single class** — a unit carries every tag that applies
(e.g. a unit can be both `buffer` and `reload-buffer`). Within the archetype
filter row, selected tags are OR'd; the row is AND'd with the other filter rows,
matching the existing Team Builder behaviour.

## Data flow

```
data/characters.json (skills.skill1/skill2/burst prose)
        │  scripts/build-archetype-tags.ts  (parse "■" blocks → match rules)
        ▼
data/archetype-tags.json  { vocabulary, tags: { slug: string[] } }
        │  import (static JSON)
        ▼
web/src/TeamBuilderPage.tsx  (archetype filter row)
```

`data/archetype-tags.json` is a **committed, generated** file in the same family
as `data/enikk-supported.json` and `data/bossing-tiers.json`. It is deliberately
kept OUT of `characters.json` because `characters.json` is fully regenerated on
every `npm run sync` — anything hand-added there would be wiped. A separate
generated file survives syncs and diffs cleanly.

## Vocabulary

Controlled, kebab-case ASCII ids. The `label`, `blurb` and `group` are stored in
the JSON's `vocabulary` block so the UI can render them without hardcoding. The
player-facing `label` carries the `▲` for buffer-family tags (e.g. `Reload ▲`);
the id stays ASCII (`reload-buffer`) because it travels through team-share URLs.

**Scope.** Stat- and damage-buffer tags match a buff on **self or allies** — a
unit that only buffs itself (e.g. privaty's self Elemental Advantage buff) still
carries the tag, because the tag answers "what does this kit buff", not "who does
it buff for". Burst-support, ally-filter and role tags stay **ally-facing** (they
are team roles, not self-buffs).

### Groups (UI filter-panel buckets)

Every tag carries a `group`; the Team Builder renders one labelled bucket per
group. This is a starter bucketing the owner can rename or rebucket by editing
the `group` on each `TagDef` (the `G.*` constants) and re-running the sweep.

| group         | what it collects                                                              |
| ------------- | ----------------------------------------------------------------------------- |
| Stat Buffs    | `▲` to core stats (ATK / DEF / HP / reload / charge / hit rate / ammo / …).   |
| Damage Buffs  | `▲` to damage types (element / core / projectile / pierce / sustained / …).   |
| Ally Filters  | Buffs that only apply to allies matching a filter (e.g. shotgun wielders).    |
| Burst Support | Team burst enablement (gauge fill, cooldown reduction, stage advance).        |
| Role          | Utility + damage profile (heal, shield, taunt, debuff, sustained, pierce, …). |

### Stat Buffs (▲ to core stats; self or ally)

| id                        | label              | means                                                               |
| ------------------------- | ------------------ | ------------------------------------------------------------------- |
| `buffer`                  | Buffer             | Buffs stats: ATK / HP / DEF / Crit / generic DMG up (self or ally). |
| `attack-buffer`           | ATK ▲              | Buffs ATK / Attack Damage (self or ally).                           |
| `def-buffer`              | DEF ▲              | Buffs DEF (self or ally).                                           |
| `hp-buffer`               | HP ▲               | Buffs Max HP (self or ally; not cover HP).                          |
| `reload-buffer`           | Reload ▲           | Buffs Reload Speed (self or ally).                                  |
| `charge-buffer`           | Charge ▲           | Buffs Charge Speed (self or ally; charge weapons).                  |
| `charge-damage-buffer`    | Charge Damage ▲    | Buffs Charge Damage (self or ally; per charge shot, not speed).     |
| `hit-rate-buffer`         | Hit Rate ▲         | Buffs Hit Rate / Accuracy (self or ally).                           |
| `ammo-buffer`             | Ammo ▲             | Buffs Max Ammunition Capacity (self or ally).                       |
| `incoming-healing-buffer` | Incoming Healing ▲ | Buffs Incoming Healing (self or ally; amplifies heals received).    |

### Damage Buffs (▲ to damage types; self or ally)

| id                   | label         | means                                                                                                  |
| -------------------- | ------------- | ------------------------------------------------------------------------------------------------------ |
| `ele-buffer`         | Element ▲     | Element-focused: buffs Elemental Advantage DMG (self or ally) **or** keys off a specific element Code. |
| `core-buffer`        | Core ▲        | Buffs core hits (Core Hit Rate / Core DMG up, self or ally).                                           |
| `projectile-buffer`  | Projectile ▲  | Buffs projectile DMG (self or ally; RL / SR / launcher kits).                                          |
| `pierce-buffer`      | Pierce ▲      | Grants/buffs Penetration (pierce), self or ally.                                                       |
| `true-damage-buffer` | True Damage ▲ | Buffs True Damage (self or ally; ignores DEF).                                                         |
| `parts-buffer`       | Parts ▲       | Buffs Damage to Parts / Interruption Parts (self or ally).                                             |
| `sustained-buffer`   | Sustained ▲   | Buffs Sustained Damage dealt (self or ally).                                                           |
| `distributed-buffer` | Distributed ▲ | Buffs Distributed Damage dealt (self or ally).                                                         |

### Ally Filters (buff allies matching a weapon/class filter)

| id                 | label             | means                          |
| ------------------ | ----------------- | ------------------------------ |
| `shotgun-buffer`   | Shotgun ▲         | Buffs shotgun-wielding allies. |
| `ar-buffer`        | Assault Rifle ▲   | Buffs assault-rifle allies.    |
| `sr-buffer`        | Sniper Rifle ▲    | Buffs sniper-rifle allies.     |
| `smg-buffer`       | Submachine Gun ▲  | Buffs submachine-gun allies.   |
| `rl-buffer`        | Rocket Launcher ▲ | Buffs rocket-launcher allies.  |
| `attacker-buffer`  | Attacker ▲        | Buffs Attacker-class allies.   |
| `defender-buffer`  | Defender ▲        | Buffs Defender-class allies.   |
| `supporter-buffer` | Supporter ▲       | Buffs Supporter-class allies.  |

These are sparse by nature — each keys off a weapon/class qualifier via
`allyQualified`, which requires the qualifier to name the _target_ of a
non-`Affects self` clause (a class keyword inside an `Affects self if …`
squad-formation condition is not a target) and the block to carry a real benefit
(a `▲` or a utility effect — heal / shield / invulnerability / cleanse / damage
reduction / burst enable):
shotgun ×7 (arcana-fortune-mate, drake, leona, neon, noir, sugar, tove), AR ×2
(ark-ranger-black, trina), SR ×2 (d-killer-wife, himeno), SMG ×1 (miranda), RL ×1
(rem), Attacker ×2 (biscuit, d), Defender ×2 (biscuit, quiry), Supporter ×1
(biscuit).
**MG** and **manufacturer** matcher tags were deliberately NOT added: no kit
buffs MG allies or allies of a specific manufacturer, so there is no prose to key
a rule off. Add one (a `TagDef` using `allyQualified`) if a future unit introduces
the pattern.

### Burst Support (ally-facing team burst enablement)

| id                   | label         | means                                                     |
| -------------------- | ------------- | --------------------------------------------------------- |
| `burst-gauge-buffer` | Burst Gauge ▲ | Fills ally Burst Gauge / buffs Burst Gauge filling speed. |
| `burst-cdr`          | Burst CDR     | Reduces ally Burst Skill cooldown.                        |
| `burst-stage`        | Burst Stage   | Re-enters / advances the Burst Stage for allies.          |

### Role (utility + damage profile)

| id                   | label              | means                                                    |
| -------------------- | ------------------ | -------------------------------------------------------- |
| `healer`             | Healer             | Recovers ally HP.                                        |
| `shield`             | Shield             | Grants allies a shield.                                  |
| `taunt`              | Taunt              | Provoke / draw enemy fire (tank).                        |
| `debuffer`           | Debuffer           | Lowers enemy DEF/ATK or raises damage taken.             |
| `damage-reduction`   | Damage Reduction   | Lowers Damage Taken by allies.                           |
| `damage-share`       | Damage Share       | Shares damage taken across allies (tank).                |
| `invulnerability`    | Invulnerability    | Grants allies invulnerability / indomitability.          |
| `cleanse`            | Cleanse            | Removes ally debuffs / grants debuff immunity.           |
| `sustained-damage`   | Sustained Damage   | Deals or enables sustained single-target damage.         |
| `distributed-damage` | Distributed Damage | Deals or enables distributed damage (split across wave). |
| `pierce`             | Pierce             | Own attacks penetrate (ignore DEF / shield-pierce).      |
| `projectile`         | Projectile         | Deals projectile explosion damage (RL / SR kits).        |

The set is **extensible**: to add a tag, append a `TagDef`
(id/label/blurb/group/test) in `scripts/build-archetype-tags.ts` and re-run the
sweep. Keep ids kebab-case ASCII, the blurb to one line, and the `group` to one
of the buckets above (or add a new `G.*` constant + group).

## How derivation works

The parser reads **only the skill prose** (`skills.skill1`, `skill2`, `burst`).
The datamined `role.*` snapshot is ignored on purpose — it carries weapon-table
columns like `reload_time`, `penetration`, `core_damage_rate` that would
false-positive (a shotgun's penetration column is not a kit pierce).

Each kit is split on the `■` block marker. Every block names its target
(`Affects all allies` / `Affects self` / `Affects all enemies` / `Affects N
enemy unit(s)`) before its effect lines, which use `▲` (buff), `▼` (debuff), and
`Deals …`. Reading the target **per block** is what separates an ally `ATK ▲`
(`buffer`) from an enemy `ATK ▼` (`debuffer`), instead of keyword-matching the
whole kit blob.

The target is read off the block's `Affects …` clause by **keyword**
(ally/enemy/self), not by matching a fixed literal — so qualified targets like
`Affects all Water Code allies`, `Affects 2 random ally unit(s)`, and `Affects
all Electric Code enemies` still classify correctly. A block is an ally block if
its `Affects …` clause contains `ally`/`allies`; `Affects the target(s)` counts
as an enemy hit unless the same clause is an ally target (`Affects the target
Defender ally`).

Two kit-text phrasings are worth knowing because they differ from the obvious:
the generic damage buff is written `Attack Damage ▲` / `Attack damage ▲` (mixed
case), not a bare `Attack ▲`; and pierce is a property a kit grants **itself**
(`Gains Pierce` / `Gain Pierce`, a self block) whereas `Pierce Damage ▲` on
allies is the pierce-buffer signal. Heal amounts are datamined with decimals
(`Recovers 4.86% of … Max HP`), so the `healer` rule treats a period followed by
a digit as part of the number, not a sentence boundary — a plain `[^.\n]` stop
dies on the decimal point and silently drops every unit whose heals are all
decimals. SR kits write the projectile-damage buff lowercase as `damage to that
projectile ▲`, so `projectile-buffer` matches case-insensitively. And an
element-focused kit may key off a specific Code in an enemy block (`Affects all
Wind Code enemies …`, `… when attacking an Electric Code target`) instead of
buffing Elemental Advantage directly — `ele-buffer` tags both shapes.

**Scoping.** A Stat Buffs / Damage Buffs tag fires on a buff in a **self or
ally** block (`buffSome`), so a unit that only buffs itself still tags. Burst
Support, Ally Filters and Role tags fire on **ally** blocks only (`debuffer` on
enemy blocks). The Ally-Filter matchers (`shotgun-buffer` and friends)
additionally require the weapon/class qualifier to name the target of a
non-`Affects self` clause, plus a real benefit in the block (a `▲` or a utility
effect such as a heal, shield or invulnerability) — so a passing mention, or a
class keyword that only appears in an `Affects self if …` squad-formation
condition (delta-ninja-thief), does not tag. Every tag carries a
`group` (one of the buckets above) that the UI uses to cluster the pills.

Rules are intentionally **high-recall heuristics** tuned for a filtering aid, not
sim-accurate kit modelling. When a rule mis-tags an edge case, fix the rule in
`build-archetype-tags.ts` (see below) — do not hand-edit the JSON, the next
sweep overwrites it.

## Re-running the sweep

```sh
npx tsx scripts/build-archetype-tags.ts            # dry run: per-tag counts + untagged list, no write
npx tsx scripts/build-archetype-tags.ts --write    # regenerate data/archetype-tags.json
```

The sweep also runs **automatically as the second half of `npm run sync`**
(`tsx src/data/sync.ts && tsx scripts/build-archetype-tags.ts --write`), so a
routine sync rewrites `data/archetype-tags.json` from the freshly-synced prose
with no extra step — it only runs if the sync itself succeeds (`&&`), so tags are
never regenerated from a half-written `characters.json`. The standalone commands
above are for re-running after a _rule_ change, or for a dry-run inspection,
without touching the DB.

The sweep is deterministic and idempotent — safe to re-run any time. Always
eyeball the dry-run counts before `--write`: a tag whose count jumps
unexpectedly usually means a kit-text phrasing changed and a rule needs a tweak.
Slugs are sorted and each tag list is ordered by vocabulary position, so the file
diffs cleanly.

After regenerating, run `npm run typecheck` (the UI imports the JSON) and
`npm run test` if the Team Builder changed.

## Tagging a single new unit (the common case going forward)

New units land one at a time via `npm run sync`, which re-runs the sweep
automatically — so the new slug is tagged the moment it syncs. The workflow:

1. `npm run sync` adds the new slug to `data/characters.json` with its kit prose
   and immediately regenerates `data/archetype-tags.json` (the sweep is chained
   onto the sync script).
2. Sanity-check the auto-tags: `npx tsx scripts/build-archetype-tags.ts`
   (dry run) — the new slug shows up under the per-tag `e.g.` examples and/or the
   `untagged` list. Confirm the tags read correctly against its kit text.
3. If a tag is wrong/missing: adjust the matching `TagDef.test` regex in
   `scripts/build-archetype-tags.ts` so it generalises (don't special-case the
   slug), then re-run `npx tsx scripts/build-archetype-tags.ts --write`.

Because the whole roster is re-derived on every sweep, there is no per-unit
state to maintain — the one-time bulk sweep and the ongoing single-unit top-ups
use the exact same command.
